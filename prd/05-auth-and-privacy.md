# 05. 認証とプライバシー

本アプリは**最初からパブリック配置・マルチユーザー前提**である。
owner による分離は将来の備えではなく、MVP 時点で有効なセキュリティ境界（[03](./03-data-model.md) §5）。

---

## 1. 認証

- **better-auth** を採用。**ソーシャル OAuth のみ**（メール+パスワードは持たない）。
  - 初手は **Google**。後で GitHub / Discord 等を追加できる作りにする（プロバイダ追加が容易）。
  - パスワードを持たない理由: 保管・リセット・総当たり対策の負担をゼロにできる。ゲーマー層には Discord/GitHub も後で有効。
- better-auth が認証テーブル（user / session / account / verification 等）を管理する。
  アプリの `owner_id` は **user.id** を指す。

## 2. データ分離（owner 境界）

- すべての**ユーザーデータ**テーブル（run / *_entry / run_image / run_payload）は `owner_id` を持ち、
  複合インデックスの先頭に置く（[03](./03-data-model.md)）。
- アプリ層は全クエリで `owner_id = セッションユーザー` を強制する。画像配信も同様（[04](./04-ingestion.md) §7）。
- **カタログはグローバル**（ゲームの客観的事実でありユーザーデータではない）。[03](./03-data-model.md) §5 の要確認事項を参照。

## 3. パブリック配置の前提

- HTTPS 必須。シークレット（`.env*`・OAuth クレデンシャル・`BETTER_AUTH_SECRET`）は**コミットしない**。
- CORS は web オリジンのみ許可。
- （将来）レート制限・濫用対策（特に Phase2 の API/MCP、Phase3 の画像アップロード）。
- 本番/開発の具体情報は公開リポに含めない（`.claude-personal/`。[02](./02-architecture.md) §9）。

## 4. プライバシー方針

- **MVP = private データのみ**。
  - 各ユーザーのデータは本人にのみ見える。**他人の個人データをピンポイントに閲覧することはできない**（当面）。
  - ランキングは**自分の run 内**のみ（[06](./06-analysis.md)）。
- **将来（Phase3）**:
  - **匿名の全体統計**（と、検討中の**横断ランキング**）に、**オプトイン**でデータを利用する。
  - そのためデータモデルに将来「**集計利用への同意フラグ（ユーザー単位）**」を足す余地を残す（MVP 未実装）。
  - 横断ランキングは**自己申告スコアの信頼性問題**（盛れる）があるため、不正対策込みで別途設計する。

## 5. Phase2 の API/MCP 認証

- ユーザー単位の**トークン（API キー）**を better-auth で発行し、API/MCP の `submit_run` 等に用いる。
- これにより、Web セッションを持たない自動投入（スクリプト・MCP クライアント）も owner に紐付く。

## 6. ロールと機能ゲート（2026-07-06）

- `user` に **`role`**（`'user'` | `'admin'`、既定 `'user'`）を追加（better-auth の `additionalFields` で
  セッションに乗せる）。管理 UI は作らず、admin 付与は DB 直接更新で十分。
- **スクショ自動解析**（[04](./04-ingestion.md) §9）は `role === 'admin'` のみ利用可。
  課金導入時は `'premium'` を enum に足すか entitlement を別途設計する（今は作らない）。
- ゲートは **API・UI の両方**に置く（非対象ユーザーにはアップロード UI 自体を出さない）。
- **worker 認証はユーザー認証と別系統**: worker 専用 API は `WORKER_API_TOKEN`（shared secret・env）で保護する
  （ユーザーのセッション/API キーは使わない。[04](./04-ingestion.md) §9.2）。
