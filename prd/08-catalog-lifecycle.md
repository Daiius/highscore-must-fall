# 08. カタログのライフサイクルと verify 手続き

> カタログ名の**正典は seed**（[`packages/database/src/catalog-data.ts`](../packages/database/src/catalog-data.ts)）。
> 本文書はその周りの**手続き**を定める。名称リスト自体はここに重複記載しない。
>
> **実装状況（2026-07-12）**: §9.1 の**機能ゲート変更**と §3 の **`evidence`**（`seed ⊆ samples` テスト含む）は
> **実装済み**。§6 の管理 UI・§7 の孤児掃除は**未実装**。

## 1. 何を解く文書か

スクショの誤読（LLM / 人間）や入力ミスから、実在しない名前がカタログに `unverified` で自動登録される。
この文書の起草時点（2026-07-11）で、これに対して次の4つが欠けていた。

1. **直す手段が無い** — カタログへの書き込み API は無く、修正は `catalog-data.ts` を手書きして再 seed するのみ。
   増える経路（自動登録）だけがあり、減る/直す経路が無い。
2. **`verified` の根拠が残らない** — `verified: true` は「開発者がスクショと突合した」という主張だが、
   **どの画像と突合したのかがコードのどこにも書かれていない**。人間の記憶にしか無い。
3. **同じ議論を再生産する** — review bot はスクショ自動解析 worker と同じモデル系統であり、
   **同じ OCR 誤読を再現する**（§8）。根拠が辿れないため、誤読指摘のたびに議論をやり直すことになる。
4. **ゲームのアップデートに開発者が追いつけない** — 新要素の裏取りには開発者自身がそこへ到達して
   撮影する必要があり、トッププレイヤーでなければ数週間遅れる。この空白期間にアプリが
   不便になってはならない（§9）。

## 2. 証拠の二層

| 層 | 実体 | 寿命 | 役割 |
|---|---|---|---|
| **恒久原典** | [`prd/samples/*.png`](./samples/)（git 管理） | リポジトリと同じ | **`verified` の唯一の根拠**。開発時・レビュー時に誰でも参照できる |
| **揮発証跡** | `run_image`（DB + blob store。[03](./03-data-model.md) §3.7） | run 削除で cascade 消滅 | 自動解析の入力・投入者本人が後から目視確認する用 |

**run のスクショは `verified` の根拠にならない。** owner スコープのユーザーデータであり、run を消せば画像も消える。
カタログはグローバル（[03](./03-data-model.md) §5）なので、消えうるユーザーデータに根拠を置けない。
→ **昇格させたい名前は、その名前が写った画像を `prd/samples/` にコミットする。**

ただしこれは「他プレイヤーの画像を原典にできない」という意味ではない。**恒久原典へ写し取れば根拠になる**
（オプトイン提供。§9.3）。閉じているのは「揮発する run_image を根拠として参照し続ける」道だけである。

## 3. `evidence`（seed の証拠フィールド）

`catalog-data.ts` の各エントリは `evidence` を持つ。

| 値 | 意味 |
|---|---|
| `'contracts-04'` 等 | `prd/samples/` の**画像ファイル名（拡張子なし）**。この画像と突合済み ＝ **verified** |
| `null` | スクショ未検証（**仮登録 / unverified**） |

- **`verified` は導出値**（`evidence !== null`）。フラグを独立に持たない。
- **DB は `evidence` を持たない。** `upgrade_catalog.verified` は seed 値の投影にすぎない（[03](./03-data-model.md) §3.5）。
  正典を1箇所に保つため、DB 側に根拠を二重管理しない。
- **テストが実在を強制する**（`database/src/__tests__/catalog-seed.test.ts`）: すべての `evidence` は
  `prd/samples/` に実在する画像を指すこと（**seed ⊆ samples**）。逆方向は制約にしない
  （画像に写っているが seed に無い名前は許す。`seed ⊆ series` と同じ非対称）。
- 複数の画像で確認できても、記録するのは**1つでよい**（根拠は1枚で足りる）。

これにより「この名前の一次情報はどれか」が**リポジトリ内で機械的に辿れる**ようになる。

## 4. カタログ行の一生

```
run 投入
  └─ 未知名 → 自動登録（unverified / kind=contract / first_seen_run_id）
                 │
                 ├─(a) 誤読・入力ミス   → マージ（alias 化）または 削除（§6・§7）
                 ├─(b) 実在するが画像なし → seed に evidence: null で仮登録 ＋ 疑義台帳に記録
                 └─(c) 実在し画像がある   → 画像を prd/samples/ にコミット ＋ evidence を書く → verified
```

判定に迷う場合の規則（2026-07-04 決定）: 矛盾・表記揺れは多数決 / 出所が1つだけなら仮登録 /
類似があればもっともらしい方を仮登録 / 読み取りミス疑いは seed に入れず疑義台帳
（`.claude-personal/catalog-suspects.md`・gitignore）へ。

## 5. 昇格手続き（正典 = seed）

**`verified` への昇格は必ずコード変更（PR）を通す。UI から verify はできない**（§6 で UI に verify ボタンを置かない理由）。

1. カタログ管理 UI で `unverified` 一覧を見る。`first_seen_run` へ辿って投入時の画像を確認する。
2. その名前が読める画像を **`prd/samples/<section>-<連番>.png` としてコミット**する
   （命名・撮影規約は [`samples/README.md`](./samples/README.md)）。
3. `catalog-data.ts` に `evidence` を書く。系統が未登録なら [`shared/src/series.ts`](../packages/shared/src/series.ts) にも足す。
4. `pnpm test`（seed ⊆ samples / seed ⊆ series）→ PR → マージ後に再 seed。

再 seed は冪等で、**`verified` を降格させない**（seed 側が仮登録でも既存の verified は触らない。`seed.ts`）。
`kind` は常に seed で上書きする（自動登録の既定 `contract` を矯正するため）。

> **なぜ UI のボタンで verify させないか。** `verified` は「開発者がこの画像を見て確認した」という主張であり、
> **レビュー可能でなければ意味がない**。DB のボタン一つで立つフラグは、後から誰も根拠を辿れず、
> §1 の問題 2・3 をそのまま残す。昇格が PR を通れば、根拠（画像）と主張（evidence）が必ず同じ diff に乗る。

## 6. カタログ管理 UI のスコープ

| やること | 内容 |
|---|---|
| 一覧 | upgrade / reward。フィルタ: `unverified` / 孤児（§7） / OU |
| 各行の情報 | 参照数（`*_entry` の件数） / `first_seen_run` へのリンク / 類似候補（[`shared/src/suggest.ts`](../packages/shared/src/suggest.ts)） |
| **マージ** | B を A に統合（配下 entry を付け替え・B の key を alias 登録・B を削除。[03](./03-data-model.md) §3.6） |
| **孤児削除** | §7 の条件を満たす行のみ。**明示実行**（自動削除しない） |
| seed スニペット | `{ name: 'X', evidence: null },` をコピーできる。開発者が `evidence` を埋めて PR にする |

| やらないこと | 理由 |
|---|---|
| verify | 正典は seed（§5） |
| `kind` の変更 | 同上 |
| 名前の直接編集 | 名前の訂正は**マージで表現する**（旧名を alias に残さないと過去 run の名寄せが壊れる） |

## 7. 孤児（orphan）の定義と掃除

**孤児** = `upgrade_catalog` / `reward_catalog` の行のうち、次を**すべて**満たすもの。

- `verified = false`
- seed に無い（`canonical_key` が `catalog-data.ts` 由来でない）
- `upgrade_entry` / `reward_entry` からの参照がゼロ
- `catalog_alias` の統合先でもない

**発生源**（どちらも「行を残す」設計になっているため必然的に溜まる）:

- run の編集で誤読名を訂正 → 旧名のカタログ行は残る。
- run の削除 → `first_seen_run_id` は `SET NULL` で、行自体は残る。

**実例**（2026-07-11 に本番から手で DELETE）: `DOUBLE-BARRELED DRONES`（正: `DOUBLE-BARRELLED`・L が2つ） /
`THIN DRONE FACTORY`（正: `TWIN DRONE FACTORY`。codex の誤読。§8 の実例でもある）。

掃除は**明示実行**とする（自動削除しない）。危険度は低いが、気づかないうちに行が消える方が調査を難しくする。

## 8. review bot のスクショ誤読への対処

**review bot とスクショ自動解析 worker は同じモデル系統である。** したがって bot は worker と**同じ誤読を再現する**。
bot がカタログ名の綴りを「スクショと違う」と指摘してきたとき、**bot 自身が画像を誤読している可能性が常にある**
（実例: PR #17 で bot が `contracts-04.png` の `TWIN DRONE FACTORY` を `THIN` と誤読して指摘した）。

議論の土俵を「画像の読み合い」から「**`evidence` があるか無いか**」に移す。

| 指摘の対象 | 対処 |
|---|---|
| **`evidence` のある名前** | **却下してよい。** 返信コメントに `prd/samples/<evidence>.png` のパスと、画像のどこ（週・行）かを書く。`evidence` の実在はテストが強制しているので、主張は検証可能な既定事実である |
| **`evidence` が無い名前**（仮登録） | **正当な可能性がある。** ただし bot の読みを鵜呑みにして seed を書き換えない。疑義台帳に記録し、画像を撮って決着させる（§5） |

反証は PR のコメントで返す。それでも同じ指摘が繰り返される場合、**コメントでの説得を続けない**。
ユーザー判断で override してマージしてよい（[`rules/commit.md`](../.claude/rules/commit.md)）。

## 9. ゲームアップデート後の空白期間

ゲームが更新されて新しいアップグレードが増えると、**開発者がその要素に到達して撮影するまで
`evidence` を書けない**（§5）。開発者はトッププレイヤーとは限らず、この空白は数週間に及びうる。
その間も他のプレイヤーは新要素を含む run を投入してくる。**この期間にアプリが不便にならないこと**を
設計の要件とする。

### 9.1 `verified` は表示専用の属性である

**`verified` が影響してよいのは、次の3つだけとする。**

1. **語句単位の表示バッジ**（run 詳細・カタログ一覧の「未検証」バッジ）
2. **カタログ管理 UI のフィルタ**（§6）
3. **裏取り済みの名前への「もしかして」の抑制**（下記のとおり、誤読への逆誘導を防ぐため）

**それ以外——投入・確定・分析・サジェストの候補プール——を `verified` で分岐させない。**
run のフローから、**機能の前提条件としての** `verified` は消える。

`verified` を機能の前提条件にすると、**開発者が追いつけない期間ちょうど、その機能が死ぬ**。
しかも §5 で昇格を PR 経由に固めたぶん、その期間は長くなる。要件と正反対に働く。

| 機能 | 旧（`verified` に依存） | 新 |
|---|---|---|
| 自動確定ゲート | 全名称が verified のとき自動確定 → **新要素を含む run は永久に draft** | **名前に関する条件を置かない**（[04](./04-ingestion.md) §9.4）。数値系の条件（error / warning / section 揃い）だけ残す |
| 類似名サジェスト | 候補プール = verified のみ → **新要素は候補に出ず、既存の似た旧名に誤誘導される** | 候補プール = **全カタログ名**。候補に verified / 未検証のバッジを出して人が選ぶ |
| 分析集計 | （もともと依存していない） | 変更なし。**未検証の語句も未分類の系統も、すべて集計に乗る**（[06](./06-analysis.md) §1.1） |

> **サジェストで禁じるのは「候補プールを絞ること」であって、「提案を出すかどうか」の抑制ではない。**
> 誤読名は unverified で自動登録されるため、**カタログには誤読が溜まる**（§9.2）。裏取り済みの名前にも
> 提案を出すと、`CLOSE SHAVE` の行に自動登録された `CL0SE SHAVE` が、`RATIONED WARHEADS` の行に
> `RATIONNED WARHEADS` が並ぶ——**正しい入力を誤読へ逆誘導する**ことになり、誤読が溜まるほど悪化する。
> したがって「入力が verified カタログ名と一致するなら提案しない」という**表示上の抑制は残す**。
> これは §9 の要件（更新直後に機能が死なない）と衝突しない: **新要素は unverified なので抑制されず、
> 提案は出続ける**。死ぬのは「すでに裏取りできている名前への提案」だけで、それは元より不要である。

> **`run.status`（draft/confirmed）と `verified` は別レイヤーである。** `confirmed` は
> 「この run の数字を分析に算入してよい」という**投入者の承認**（分析集計は confirmed のみ対象）。
> `verified` は「この名前がゲームに実在する」という**開発者の裏取り**。run をいくら確定しても
> カタログは verified にならないし、その逆も要らない。

### 9.2 誤読が confirmed に入ることを受け入れる

名前の条件を外すと、**LLM が誤読した名前がそのまま confirmed の run に入る**（実例: `THIN DRONE FACTORY`）。
これは**受け入れる**。根拠は2つ。

**(1) 数値の誤読は分析を直撃するが、名前の誤読は分析を歪めない。** `final_score` が違えば散布図が壊れるので、
数値系のゲート（error / warning / section 揃い）は残す。一方、名前を1つ読み違えても系統構成のバーが
1本ぶんズレるだけで、**ゲームのランダム性やプレイヤーの調子による分散に埋もれる**。
分析に 100% の精度は求めない（[06](./06-analysis.md) §1.1）。

**(2) 回収の網が後段に残る。** 誤読名は `evidence` を持てないので **seed に入らず `unverified` のまま**で、
語句単位の「未検証」バッジが付き、管理 UI の未検証一覧に出続け、マージ・孤児掃除の対象であり続ける（§6・§7）。
訂正すれば分析へ反映される（現状は毎回 SQL。将来キャッシュを入れる場合も**訂正が反映される経路は必須**。
[06](./06-analysis.md) §4.1）。

確定を厳しくして止めるより、**確定を速くして訂正を安くする**方を取る。
これは「**時間とともに信頼性が上がるシステム**」（[06](./06-analysis.md) §1.1）の一部である——
アップデート直後は知らない contract が混ざってよく、投入とスクショの蓄積につれて静かに正確になっていく。

### 9.3 他プレイヤーのスクショを原典に採用する道（将来）

開発者が到達できない要素は、**先に到達した他プレイヤーのスクショ**でしか裏取りできない。
§2 の原則（run のスクショは揮発するので根拠にならない）は維持したまま、次の道を将来開ける:

投入 UI に「**この画像をカタログの一次情報として提供する**」オプトインを置き（`run_image` に
提供フラグ）、チェックされた画像だけ admin が管理 UI からダウンロードして `prd/samples/` に
コミットできるようにする。恒久原典はあくまでリポジトリ側に写し取る。

**実装しない**（当面 admin 単独運用で実害がない）。「verified の原典はリポジトリの画像だけ」と
言い切るとこの道が閉じるため、構想として残す。プライバシー方針は [05](./05-auth-and-privacy.md) に従う
（無断で他人の画像をリポジトリに公開しない＝オプトインが要る理由）。
