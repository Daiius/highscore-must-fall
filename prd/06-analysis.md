# 06. 分析機能

本プロジェクトの動機は「どのアップグレードをどのタイミングで取ったか」「ハイスコアの傾向」を見ることである。
ただし**統計的（推論的）分析は Phase2**とし、MVP はまず**記述的分析**で「確実に取り込め、閲覧でき、傾向を眺められる」状態を作る。

データ構造は [03](./03-data-model.md)、用語は [01](./01-game-domain.md) を参照。

---

## 1. MVP（記述的）

### 1.1 基盤

- **run 一覧**: フィルタ/ソート（`played_at`・`final_score` 等）。
- **run 詳細**: コア指標 / アップグレード履歴（週・順序・reroll を可視化）/ リワード台帳 / 添付スクショの閲覧・編集・削除。
- **自己ベスト・自分の run 内ランキング**: `final_score` 等の上位。

#### 精度の方針: 全データを乗せ、後から直す

**分析に 100% の精度を求めない。** スコアはゲームのランダム性・プレイヤースキル・その日の調子といった
**分析対象外の要因で大きく変動する**。語句の読み違いが1つ混ざって系統構成のバーが1本ぶんズレることの影響は、
その分散に埋もれる。

したがって、**集計対象を「正しいと確認できたもの」に絞らない**。

- **未検証（`unverified`）の語句も集計に乗せる。** 除外しない。
- **未分類（`unknown`）の系統も集計に乗せる。** 「未分類」を一級のバケットとして表示する。
- 曖昧さは**語句単位の「未検証」バッジで明示する**にとどめる（分析結果に多少の揺れがありうる、という通知）。
- 語句や系統は**後から直る前提**で扱う。現状の集計はキャッシュせず毎回 SQL で組み立てるため（§4）、
  **直せば次の描画から反映される**。分析結果が後から少し変わることを許容する。

**目指すのは「時間とともに信頼性が上がるシステム」である。** ゲームのアップデート直後は知らない contract が
混ざっていてよい。投入が進み、スクショが集まり、語句が verify され、系統が分類されるにつれて、
**同じデータの分析結果が静かに正確になっていく**。この単調改善を止めないことが設計の要点であり、
逆に「正しいと確認できるまで集計に入れない」設計は、精度を上げるどころか**アップデート直後に分析を空にする**。

→ この方針の帰結として、`verified` は**表示専用の属性**であり、投入・確定・分析・サジェストを分岐させない
（[08](./08-catalog-lifecycle.md) §9.1）。

### 1.2 記述的分析ビュー

- **スコア散布図**: `final_score` × `played_at` の**散布図**（ゲームにランダム性があり、
  run は 1日1回でもないため折れ線にしない）。横軸は**実時間 ⇄ 順番（run を等間隔に詰める）**を
  切替可能（日が空いた際の不自然な横伸び対策。タイムラインと共通トグル）。
- **アップグレード取得タイムライン**（対象は直近 N run。上限は UI に明示。2モード）:
  - **系統構成（既定）**: run ごとのカードに**週×系統の取得数を積み上げ棒**で表示。
    並び順は**時系列 ⇄ スコア順**を切替（「どの要素がスコアを伸ばすか」の目視分析には
    日付軸は不要 — スコア順に並べて構成の勾配を読む）。カードは**縦1列**（読み順を一意にする）、
    左端アクセントは**スコア低→高＝暗→明の indigo 単一 hue ランプ**（散布図のスコア点と同系統。
    validator 検証済み）。週軸は W1〜全 run の最大週にゼロ埋め、Y 上限も全カード共通
    （カード間で取得タイミング・積み上げ高さを比較可能にするため）。
    分類は二次情報由来の暫定マッピング
    （`shared/src/series.ts`、[01](./01-game-domain.md) §3）。カタログ属性への昇格は Phase2 で判断。
  - **アップグレード別**: y=カタログ名（取得頻度降順）× x=run のドットマトリクス。
    **色 = 系統**（主砲=緑 / 核=黄 / シールド=青 / フレイル=紫 / 自動防衛=赤 /
    OU・未分類=保留につき無彩色。配色は validator 検証済み）。週はツールチップ。
- ~~アップグレード頻度~~・~~取得タイミング分布~~: **削除済み**（取得タイムラインが役割を兼ねるため。
  頻度はほぼ全て1回でランキングの意味が薄かった）。
- 上部の集計値は**確定ラン数・ベストスコア**のみ（平均スコアは下手な時期も均されて意味が薄いため出さない）。

### 1.3 最小カタログ管理（記述分析の前提条件）

- 名寄せが崩れると頻度・タイミング分布が壊れるため、MVP に**最小限**のカタログ管理を含める。
- 操作は3つ:
  - **未検証（unverified）一覧**の表示。
  - **verify** マーク。
  - 既存エントリへの**エイリアス統合（マージ）**（[03](./03-data-model.md) §3.6）。
- リッチ属性（カテゴリ・色・レアリティ）は Phase2。

## 2. Phase2（推論的・本命）

- **アップグレード × タイミング → スコア相関**:
  例「WEEK1 で X を取った run の平均 `final_score`」 vs「取らなかった run」。
- **リロール分析**: リロール回数（`reroll_count`）× スコア。
- **リワード分析**: どの reward が点・頻度に効くか。
- ※ これらは**数十〜数百 run 蓄積後**に意味を持つ。MVP の集計・グラフ部品を再利用して実装する。
- カタログのリッチ属性（カテゴリ別の傾向など）もここで活用。

## 3. Phase3（横断・要オプトイン）

- 匿名の**全体統計**、検討中の**横断ランキング**（[05](./05-auth-and-privacy.md) §4）。
- 自己申告スコアの信頼性対策が前提。

## 4. 実装メモ

- 集計は素のクエリで（事前集計テーブルは作らない。[03](./03-data-model.md) §6）。実測で遅ければ導入を検討する。
- グラフ描画ライブラリは実装時に選定（軽量な可視化で十分）。

### 4.1 将来キャッシュ／事前集計を入れるときの条件

分析が重くなればキャッシュや事前集計を入れてよい。ただし §1.1 の「**時間とともに信頼性が上がる**」を
壊してはならない。具体的には、**語句のマージ・訂正・系統分類の追加が、いずれ分析結果へ反映されること**を
保証する経路が要る。満たし方は問わない（実装時に選ぶ）:

- 訂正イベントで**該当キャッシュを無効化**する、または
- 古いデータは更新せず、**パフォーマンスに影響しない範囲で漸進的に再集計**する（遅れて正確になるのは許容。
  §1.1 の思想と整合する）。

**やってはいけないのは、訂正が永久に反映されない集計を作ること**（＝誤読が凍結される）。
