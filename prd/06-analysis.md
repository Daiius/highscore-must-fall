# 06. 分析機能

本プロジェクトの動機は「どのアップグレードをどのタイミングで取ったか」「ハイスコアの傾向」を見ることである。
ただし**統計的（推論的）分析は Phase2**とし、MVP はまず**記述的分析**で「確実に取り込め、閲覧でき、傾向を眺められる」状態を作る。

データ構造は [03](./03-data-model.md)、用語は [01](./01-game-domain.md) を参照。

---

## 1. MVP（記述的）

### 1.1 基盤

- **run 一覧**: フィルタ/ソート（`played_at`・`final_score` 等）。
- **run 詳細**: コア指標 / アップグレード履歴（週・順序・reroll を可視化）/ リワード台帳 / 添付スクショの閲覧・編集・削除。
- **自己ベスト・自分の run 内ランキング**: `final_score` 等の上位。

### 1.2 記述的分析ビュー

- **スコア推移**: `final_score` × `played_at` の折れ線（腕前の推移）。
- **アップグレード頻度**: 全体／週別の取得回数ランキング（`upgrade_entry` を `upgrade_catalog_id` で集計、`entry_type=upgrade` のみ）。
- **アップグレード取得タイミング分布**: 特定アップグレードが**何週・何手目（`upgrade_order`）で取られがちか**の分布。

### 1.3 最小カタログ管理（記述分析の前提条件）

- 名寄せが崩れると頻度・タイミング分布が壊れるため、MVP に**最小限**のカタログ管理を含める。
- 操作は3つ:
  - **未検証（unverified）一覧**の表示。
  - **verify** マーク。
  - 既存エントリへの**エイリアス統合（マージ）**（[03](./03-data-model.md) §3.6）。
- リッチ属性（カテゴリ・色・レアリティ）は Phase2。

## 2. Phase2（推論的・本命）

- **アップグレード × タイミング → スコア相関**:
  例「WEEK1 で X を取った run の平均 `final_score`」 vs「取らなかった run」。
- **リロール分析**: リロール回数（`reroll_count`）× スコア。
- **リワード分析**: どの reward が点・頻度に効くか。
- ※ これらは**数十〜数百 run 蓄積後**に意味を持つ。MVP の集計・グラフ部品を再利用して実装する。
- カタログのリッチ属性（カテゴリ別の傾向など）もここで活用。

## 3. Phase3（横断・要オプトイン）

- 匿名の**全体統計**、検討中の**横断ランキング**（[05](./05-auth-and-privacy.md) §4）。
- 自己申告スコアの信頼性対策が前提。

## 4. 実装メモ

- 集計は素のクエリで（事前集計テーブルは作らない。[03](./03-data-model.md) §6）。
- グラフ描画ライブラリは実装時に選定（軽量な可視化で十分）。
